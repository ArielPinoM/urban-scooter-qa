# US-71: La documentación de DELETE /api/v1/courier/:id indica incorrectamente que el id se envía en el body en lugar de ser un path parameter

# Detalles clave

## Severidad
🟠 Major

## Prioridad
🟧 High

## Entorno

- Documentación API (apidoc)

- Postman 12.16.4

## Componente
API - Eliminar un mensajero

## Descripción

La especificación del endpoint `DELETE /api/v1/courier/:id` en la documentación generada con apidoc muestra que el parámetro `id` se debe enviar en el cuerpo de la solicitud (JSON):

```json
{
  "id": "3"
}
```

Sin embargo, el endpoint está implementado correctamente como un path parameter (`/api/v1/courier/:id`), donde el `id` debe ir en la URL (ej. `/api/v1/courier/3`). Si se sigue la documentación al pie de la letra y se envía el id solo en el body, la solicitud falla porque la ruta no incluye el identificador.

La documentación debe reflejar fielmente el contrato real de la API.

### Precondiciones

- Servidor en ejecución.

- Existe un repartidor con `id` conocido.

### Pasos para reproducir

1. Seguir la documentación: enviar `DELETE /api/v1/courier/` (sin id en la URL) con el body { "id": "<id_mensajero>" }.

2. Observar el resultado.

3. Comparar con el comportamiento real correcto: enviar `DELETE /api/v1/courier/<id_mensajero>` (sin body o con body vacío).

### Resultado esperado (según documentación)

- La documentación debería especificar que el `id` es un parámetro de ruta (parte de la URL) y no un campo del cuerpo.

- El ejemplo debería mostrar la URL con el id y sin body, o con un body vacío.

### Resultado actual (documentación actual)

- La tabla de parámetros describe `id` como un campo del cuerpo.

- El ejemplo de solicitud incluye `{ "id": "3" }` en el body.

- La URL en el título sí muestra `:id`, pero la descripción induce a error.
